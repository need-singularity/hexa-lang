## Bugs

### #6 `len()` on a string returned the first byte

- Root cause: `mn_load_off(base, 0)` reads the array-length prefix that
  arrays have, but C-strings have no such prefix — so offset 0 is just
  the first character byte.
- Repro: `let s = "42"; return len(s)` → 52 (ASCII '4'), not 2.
- Fix: track string-typed vars (`poc_var_str_names/lens`) bound from
  `let s = "lit"` so `len()` can return the compile-time length.
  `StringLit` args also get an immediate length.

### #4 struct return-type propagation missing

- Root cause: `let t = make_tok(...)` lost the struct type — caller had
  no way to resolve `t.field` since `poc_var_struct_of` returned "".
  Printed `[poc] unknown struct for var: t` and emitted wrong code.
- Fix: `poc_fn_ret_structs` registry; `poc_lower_program` registers
  each `FnDecl`'s struct return type; the `let` handler binds
  `stmt.name` as that struct when RHS is a `Call` to a known fn.

### #5 regalloc: fresh-temp overflow to XZR

- Root cause: callee-saved window `x19..x28` = 10 vids. Nested
  `dx*dx + dy*dy` with 2 struct params burned 9 vids; final add
  landed at vid 29, which `mn_regfor` maps to x31 (XZR) → silently
  zero.
- Repro: classic `dist_sq` with 2 struct params returned 0 instead
  of `dx² + dy²`.
- Fix: `poc_pick_binop_result_vid(lv, rv, before)` — if either operand
  is a fresh temp (vid ≥ `before` and ≤ 28), reuse its slot as the
  result slot (dead after the op). Falls back to `poc_fresh_vid()`
  when both operands are persistent bindings.
