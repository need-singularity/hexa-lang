# 2026-04-06: VM Performance 333x Breakthrough

## Summary
16-round systematic optimization session: interpreter 2.00s → JIT 0.006s (333x).
Tiered execution architecture (JIT → VM → Interpreter) with automatic mode selection.

## Changes

### Performance (vm.rs)
- Globals: HashMap → Vec<Value> O(1) indexed lookup
- All arithmetic/comparison/bitwise/unary: in-place stack modification (no pop+push)
- GetLocal/Const: Int-first branch prediction
- JumpIfFalse/True: peek + set_len (skip Value move)
- CallFn: ptr::copy_nonoverlapping for args
- Code pointer: raw *const OpCode cache (eliminates code_segments indexing)
- Return sentinel: removes ip >= code_len branch per dispatch
- #[cold] error paths, #[inline(always)] dispatch loop

### Value Size (env.rs + 13 files)
- 72 bytes → 32 bytes (56% reduction)
- Boxed: Fn, Lambda, Map, Intent, EnumVariant, EffectRequest, TraitObject
- Added TraitObjectInner struct
- L1d cache density 2.25x improvement

### Tiered Execution (main.rs)
- Default path: JIT (Cranelift) → VM (bytecode) → Interpreter (tree-walk)
- JIT/VM skip type-checking and ownership analysis (deferred to interpreter)
- VM reuses parsed AST (no double-parse)

### New Features
- `-e` / `--eval` flag: inline code execution (`./hexa -e "println(42)"`)
- Dream engine stack safety: mutations run in 2MB-stack thread

### Bug Fixes
- dream::tests::test_dream_engine_fibonacci stack overflow (pre-existing)

## Key Learnings
1. Adding OpCode enum variants degrades LLVM jump table — fused opcodes reverted
2. Value size (72→32B) gave measurable cache improvement
3. Match arm reordering doesn't help — LLVM optimizes regardless
4. unsafe push_fast + reserve hurts — Vec::push is already optimal in release
5. PGO requires profile/code version match
